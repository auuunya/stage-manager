# references/best_practice.md

> 指导任务拆解、验收、evidence 编写与日志质量。字段 schema 以 `stage_template.md` 为准，本文件仅补充最佳实践。

## 1. 核心哲学

- **隔离原则**：Stage N 的失败不应导致全局崩溃，需定义容错或回滚方案。
- **文档即代码**：像维护代码一样维护 `stage-XXX.md`，保持状态实时同步。
- **增量更新**：优先只更新受影响的 section，不整体重写。

---

## 2. 任务拆解 (INVEST)

| 准则            | 说明                               | 反面案例                 | 正面案例                          |
| :-------------- | :--------------------------------- | :----------------------- | :-------------------------------- |
| **Independent** | 任务间解耦，无循环依赖             | "开发接口并等待 UI 确认" | "完成 API 接口定义与 Mock 实现"   |
| **Valuable**    | 每个任务产出可感知的价值           | "研究数据库文档"         | "输出数据库表结构设计文档"        |
| **Estimatable** | 单任务 < 2 天                      | "完成底层架构搭建"       | "实现基础日志模块与异常捕获"      |
| **Small**       | 原子化，便于勾选                   | "编写并测试所有脚本"     | "编写脚本 A；通过脚本 A 压力测试" |
| **Testable**    | 有明确验收动作                     | "优化性能"               | "将响应延迟从 500ms 降至 200ms"   |

- 无法 1~2 天完成则继续拆分；无可验证结果则重写描述。
- `acceptance` 显式指向验收项；`deliverables` 与 `evidence` 不混用。

---

## 3. 验收标准编写

优先使用"动作 + 结果"，禁止模糊表达（"符合预期""基本完成"）。

| 类型       | 示例                                                  |
| :--------- | :---------------------------------------------------- |
| 技术类     | 代码已 Push，编译通过，核心单测通过                   |
| 文档类     | README 已更新，接口文档已同步                         |
| 部署类     | 测试环境成功运行，无 P0 Bug                           |
| 性能类     | 明确负载下达到指标门槛，保留测试结果                  |
| 可追踪类   | 日志、ADR、阶段总结已同步                             |

`required_checks` 写可验证检查点（如 `login_api_returns_token`、`100_rps_no_error`）。

### evidence 约定

- 代码类：保持项目原始路径（`src/...`、`tests/...`）
- 非代码类：统一放 `.stage/` 下（`reports/`、`docs/`、`logs/`、`snapshots/`）
- 每个 P0 任务至少一份可读摘要 evidence；每个 AC 至少一条可核验 evidence
- 实施型阶段：仅清单/总结/ADR 不构成完成证据

---

## 4. 风险矩阵

优先考虑技术、环境、时间三类风险。

> 示例：风险=第三方 API 并发限制 → 触发=用户超 100 → 应对=令牌桶限流+重试

- 暂无明确风险时保留表头即可，不编造。高风险项补充回滚/降级方案。

---

## 5. 日志与提交规范

日志记录关键决策，非琐碎动作。禁止"优化了代码""做了一些调整"等模糊表述。

- 日志需指向具体模块、接口、风险项或 ADR
- Git Commit 涉及架构决策时包含 ADRS ID，如 `feat: 实现加密存储 [ADRS-005]`

---

## 6. 命名约定

- **文件**：`stage-<NNN>-<英文短名>.md`（如 `stage-003-stripe-integration.md`）
- **标题**：`# STAGE-<NNN>: <name>`，与 frontmatter 对齐
- **编号**：统一三位数字 — `TARGET-001`、`TASK-001`、`AC-001`、`LOG-001`、`ADRS-001`、`SUMMARY-001`
