# references/best_practice.md

## 1. 核心哲学：原子化与闭环

阶段计划的核心目的不是“记录”，而是“驱动执行”。每一个阶段文档必须是一个**可交付的任务闭环**。

- **隔离原则**：Stage N 的失败不应导致全局崩溃，必须在 Stage N 中定义容错或回滚方案。
- **文档即代码**：像维护代码一样维护 `stage-XX.md`，保持状态实时同步。

---

## 2. 任务拆解准则 (The I.N.V.E.S.T. Rule)

在填充 `## 4. 任务拆解` 时，所有任务必须符合以下标准：

| 准则            | 说明                                   | 反面案例                 | 正面案例                          |
| --------------- | -------------------------------------- | ------------------------ | --------------------------------- |
| **Independent** | 任务间尽量解耦，不产生复杂的循环依赖。 | “开发接口并等待 UI 确认” | “完成 API 接口定义与 Mock 实现”   |
| **Valuable**    | 每个任务完成后，必须产出可感知的价值。 | “研究数据库文档”         | “输出数据库表结构设计文档”        |
| **Estimatable** | 颗粒度适中，单个任务执行时间应 < 2天。 | “完成底层架构搭建”       | “实现基础日志模块与异常捕获”      |
| **Small**       | 原子化操作，便于勾选进度。             | “编写并测试所有脚本”     | “编写脚本 A；通过脚本 A 压力测试” |
| **Testable**    | 必须有明确的验收动作。                 | “优化性能”               | “将响应延迟从 500ms 降低至 200ms” |

---

## 3. 验收标准 (Definition of Done) 编写指南

不要写模糊的描述，使用“动作 + 结果”的组合。

- **技术类**：代码已 Push，单测覆盖率 > 80%，通过编译检查。
- **文档类**：README 已更新，接口 API 文档已同步，Swagger 可访问。
- **部署类**：在测试环境成功运行，无 P0 级 Bug。

---

## 4. 风险与应对 (Risk Matrix) 示例

AI 在填写此部分时，应至少覆盖 **技术风险、环境风险、时间风险**。

> **示例：**
>
> - **风险**：第三方 API 并发限制。
> - **触发条件**：测试用户超过 100 人时。
> - **应对**：实现令牌桶限流算法，并增加请求重试机制。

---

## 5. 进度日志与 Git 提交规范

日志与代码提交应记录**关键决策**而非琐碎动作。

- **✅ 日志写法**: 完成 `auth` 模块重构。[决策] 采用 Session 方案，详见 **[ADRS-002]**。
- **✅ Git 提交规范**: 凡是涉及架构决策的代码修改，Commit Message 必须包含对应的 ADRS ID。
  - _示例_: `feat: 实现加密存储逻辑 [ADRS-005]`

---

## 6. 命名约定 (Naming Convention)

- **文件命名**：`stage-<序号>-<英文短名>.md` (序号建议使用三位数字填充，如 001)。
- _Good_: `stage-003-stripe-integration.md`
- _Bad_: `Stage3_FixBug.md`

---

## 7. 结项检查清单 (Stage Closure Checklist)

在将 `STAGES.md` 中的状态改为 `done` 前，请确保：

1. [ ] 所有 **[P0]** 任务已勾选。
2. [ ] **## 5. 验收标准** (DoD) 100% 通过。
3. [ ] `## 3. 非范围` 中产生的新想法已移至 `BACKLOGS.md` 或记录在下一个 Stage。
4. [ ] 执行 `stage:summary` 保存最后记忆快照。
